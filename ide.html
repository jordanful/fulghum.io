I Prototyped an IDE for How We Actually Code Now

Like most devs, I've spent the last six months getting used to this new AI-assisted coding world. While vibecoding is not only exciting but now my default way of working, I have found that the tools that have embraced it fall short in various ways.

The problem with current tools is they're all compromised. Cursor bolts AI onto VS Code's legacy architecture. You can feel the weight of decades of assumptions about how editors should work and it feels like the core loop is crammed into a third of the window. Claude Code gets closer to the conversational model but strips away too much context so it feels like coding through a keyhole. The newer tools like Codex optimize for the wrong cycle time, focusing on pull requests when most of my iterations are immediate and focused. Replit gets a lot of things right, simplifying or abstracting critical technical pieces of the application like a database, but the LLM layer feels isolated away as another abstraction as opposed to the thing. The engine itself.

What I realized is that we've been thinking about this backwards. Instead of adding AI to traditional IDEs (or a chat interface to a terminal), we need to start with the conversational loop and built around it.

So I prototyped something different. A native Mac app that's an IDE designed around the reality that most application development today happens in conversation with an LLM. Not as an afterthought or a plugin, but as the primary interface. The core insight is that modern programming isn't really about managing files and directories but rather managing context, intent, and tasks. You're constantly shuttling between "show me this," "change that," "what happens if," and "make it work."

The atomic unit isn't a file or a projectâ€”it's a task. Think of it like a feature branch, but richer. You describe what you want to build next, and the IDE creates a living implementation plan in markdown that updates as you work. It writes and maintains tests automatically. Multiple tasks can run in parallel, each with their own context and state. Deployments to production environments are tied 1:1 with these AI-led tasks, so reverting or undoing is 1-click simple.

Ok so what's the core loop?

  1. Describe what you want
  2. The agent creates an implementation plan via markdown. It will follow this through its development of the feature
  3. If needed, the agent will add or update any tests based on the changes
  4. Before marking it as ready to review, the test suite runs automatically
  5. The human ultimately approves the change, and doing so creates a new version of the app that can be deployed to production

Multiple tasks can be run in parallel and monitored individually.

The rest of the interface reflects this task-centric thinking. The primary window is a preview. If you're building for the web like me, visual iteration is the core loop. There's no separation between "writing code" and "talking to AI" and "seeing the thing". Context flows naturally between what you're looking at, what you're thinking about, and what you want to happen next.

Getting started isn't staring at a blank file. You pick from templates - SaaS app, ecommerce store, REST API, blog platform - or just describe what you want in plain English of course. The IDE understands the shape of what you're building and sets up the scaffolding accordingly, suggesting the right stack for the job.

There's a data explorer and admin panel built in natively (I always loved this about Django). You can poke around your database, visualize data, and even ask the built-in chatbot questions about your schema data itself. It's like having a data analyst in your dev environment.

I suspect a lot of non-technical vibecoder get caught up on environment management (the whole "guys check out this app I just made: http://localhost:3000" meme). The local-to-production workflow in my prototype is really simple. Think Stripe's test mode toggle, but for your app. One switch, and you're looking at live data, live users, live everything. Switch back to local safety with a click.

Instead of the traditional write-compile-debug cycle, you get something more fluid. More like sketching ideas and refining them in real time. The computer becomes a genuine collaborator rather than just a fancy typewriter.

I kept thinking about Coda, which had this beautiful simplicity where everything just worked together from CSS to HTML to the preview itself. This is kinda like Coda's spiritual successor for the age of vibecoding.

This is still rough around the edges and just a prototype. Lots of assumptions baked in that probably don't hold for different kinds of development. But it was fun to build, and I learned something about what I actually want from my tools.
